# Evaluation Criteria

This note captures the course evaluation criteria shared in class and turns them into an explicit review frame for the ZTBrowser AIDev artifact package.

## Core criterion

The main criterion is methodological consistency across the full delivery chain:

- specification
- tests
- code in GitHub
- working application

The specification is expected to include:

- user stories
- use cases
- requirements
- a BDD approach
- architectural description by levels

A reviewer should be able to inspect the project through this whole framework and see that the parts are aligned with the methodology discussed in class.

## Required artifacts

To score well, the project must provide:

- business and technical specification
- tests
- code in GitHub
- a working application

For this repo, that means the course artifacts must stay consistent with the actual ZTBrowser implementation and documented architecture rather than becoming a detached paperwork layer.

## Grading scale

### 10
Specification (Business + Tech), tests, code in GitHub, and a working application are all complete, consistent, and fully follow the methodology.

### 9
Specification (Business + Tech), tests, code in GitHub, and a working application are present with minor gaps or inconsistencies.

### 8
Specification (Business + Tech), tests, code in GitHub, and a working application are present, but some parts are incomplete or not fully aligned.

### 7
Specification (Business + Tech) and code in GitHub are present, tests are weak or incomplete, application works partially.

### 6
Specification (Business + Tech) is incomplete, tests are minimal or missing, code in GitHub is inconsistent, application barely works.

### 5
Specification (Business + Tech) is weak or fragmented, tests are missing, code in GitHub is poor, application mostly does not work.

### 4
Specification (Business + Tech) is mostly missing, tests are missing, code in GitHub is incomplete, application does not work properly.

### 3
Only some elements are present, such as code in GitHub or a partial specification, with no consistency and a non-functional application.

### 2
Minimal submission with most elements missing: specification, tests, code in GitHub, and application.

### 1
No meaningful specification, tests, code in GitHub, or working application are provided.

## Practical implication for ZTBrowser

The target is not merely to produce files. The package should let a reviewer trace one coherent chain:

1. business and technical framing
2. user stories, use cases, BDD scenarios, and requirements
3. tests and traceability
4. implementation in GitHub
5. working application and deployment evidence

The highest-risk failure mode is inconsistency between these layers. That is the main thing this course context must prevent.
