# Validation Story and Landing Copy

## Validation story

ZTBrowser does not yet claim broad user adoption. The available validation evidence is technical and product-structural:

- the repo proves browser-side verification is implemented
- the canonical enclave repo proves release provenance and rebuild discipline
- the AWS Nitro path proves the architecture works against a real trusted execution environment
- Micrus proves the concept can extend into an application scenario, not just a toy protocol demo

This is early-stage validation for a developer trust platform, not market validation from production revenue.

## Recommended landing-page headline

**Prove enclave-backed services to the browser, not just to your backend.**

## Recommended subheadline

ZTBrowser verifies attestation in the browser, maps measured identity to public release provenance, and helps teams deploy attested services with a repeatable operator flow.

## Core proof points

- browser-side verification of attestation documents
- public facts mapping for workload provenance
- canonical enclave release artifacts and rebuild verification
- real AWS Nitro deployment path
- demo/self-signed path for education and rapid iteration

## CTA concepts

- Verify a live attested service
- See how canonical release provenance maps to runtime proof
- Run a deployment with `ztdeploy`

## What not to say

- do not claim multi-cloud support today
- do not claim CoCo integration today
- do not claim FHE support today
- do not imply the facts node itself is the trust root
